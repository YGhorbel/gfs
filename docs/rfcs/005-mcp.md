# RFC 008 — MCP server for GFS / data-plane

## Overview

The **Model Context Protocol (MCP)** server exposes GFS and data-plane capabilities as **tools** so that AI assistants (e.g. Cursor, Claude Desktop) and other MCP clients can perform repository and compute operations without invoking the CLI directly. The server reuses the same domain use cases and ports as the CLI; the MCP layer is an **adapter** that translates tool invocations into use-case calls and returns structured results.

This RFC defines the **scope**, **tool set**, **behaviour**, and **integration** of the MCP server. Implementation follows the hexagonal architecture: domain use cases and ports remain unchanged; the MCP server is a thin adapter (similar to the CLI) that resolves context (e.g. repo path), calls use cases, and maps outcomes to MCP tool results.

---

## Goals

- **Tool-based access**: Expose a consistent set of tools (list providers, status, commit, log, checkout, init, compute lifecycle, etc.) that mirror CLI capabilities.
- **Single repo context**: Tools operate on a configurable repository path (default: current working directory when the server starts, or a path supplied via server config).
- **Structured output**: Tool results are machine-readable (e.g. JSON) so that AI clients can reason over status, commit ids, and errors.
- **No duplicate logic**: All behaviour is implemented in the domain; the MCP server only parses arguments, calls use cases, and formats responses.

---

## Tool set

The MCP server SHALL expose the following tools. Each tool’s arguments and return shape align with the corresponding CLI command and RFCs (RFC 006 — CLI).

### 1. `list_providers`

Lists supported database providers and their versions (and optionally features). Equivalent to `gfs providers` (see [006-cli-providers](006-cli/006-cli-providers.md)).

| Argument | Type   | Required | Description |
|----------|--------|----------|-------------|
| *(none)* | —      | —        | No arguments. |

**Returns**: A list of providers with at least `database_provider`, `versions`, and optionally `features`. Format may be a table or JSON array.

---

### 2. `status`

Returns the current state of the GFS repository and its compute instance. Equivalent to `gfs status` (see [006-cli-status](006-cli/006-cli-status.md)).

| Argument | Type   | Required | Description |
|----------|--------|----------|-------------|
| `path`   | string | no       | Path to the GFS repository root. Defaults to server-configured repo path or cwd. |

**Returns**: Status payload with `repository` (e.g. `current_branch`) and `compute` (e.g. `provider`, `version`, `container_status`, `container_id`, `connection_string`) when applicable. JSON object.

---

### 3. `commit`

Creates a new commit in the repository. Equivalent to `gfs commit -m <message>` (see [006-cli-commit](006-cli/006-cli-commit.md)).

| Argument         | Type   | Required | Description |
|------------------|--------|----------|-------------|
| `message`        | string | **yes**  | Commit message. Must be non-empty. |
| `path`           | string | no       | Path to the GFS repository root. |
| `author`         | string | no       | Override author name. |
| `author_email`   | string | no       | Override author email. |

**Returns**: Commit result (e.g. commit id, branch name). JSON object. Errors (e.g. empty message, repo not found) as structured error.

---

### 4. `log`

Returns commit history from HEAD or a given revision. Equivalent to `gfs log` (see [006-cli-log](006-cli/006-cli-log.md)).

| Argument   | Type   | Required | Description |
|------------|--------|----------|-------------|
| `path`     | string | no       | Path to the GFS repository root. |
| `max_count`| number | no       | Maximum number of commits to return. |
| `from`     | string | no       | Start at this revision (branch or commit hash). |
| `until`    | string | no       | Stop before this revision (exclusive). |

**Returns**: List of commits (id, message, author, timestamp, etc.). JSON array.

---

### 5. `checkout`

Switches the active branch or commit (HEAD) and updates the workspace. Equivalent to `gfs checkout` (see [006-cli-checkout](006-cli/006-cli-checkout.md)).

| Argument   | Type   | Required | Description |
|------------|--------|----------|-------------|
| `revision` | string | **yes**  | Branch name or full 64-char commit hash. |
| `path`     | string | no       | Path to the GFS repository root. |
| `create_branch` | string | no | When set, create a new branch with this name at `revision` and switch to it (like `gfs checkout -b <name> [<start>]`). |

**Returns**: Success with new HEAD (branch or commit). JSON object. Errors (e.g. branch not found, no commits) as structured error.

---

### 6. `init`

Initializes a new GFS repository at the given path. Equivalent to `gfs init` (see CLI init command).

| Argument            | Type   | Required | Description |
|---------------------|--------|----------|-------------|
| `path`              | string | no       | Directory where to create `.gfs`. Defaults to server-configured path or cwd. |
| `database_provider` | string | no       | If set, provision and start the database (e.g. `postgres`, `mysql`) after creating the repo. |

**Returns**: Success with repo path and optionally compute info. JSON object.

---

### 7. `compute`

Performs compute lifecycle or inspection actions. Equivalent to `gfs compute <action>` (see [006-cli-compute](006-cli/006-cli-compute.md)).

| Argument | Type   | Required | Description |
|----------|--------|----------|-------------|
| `action` | string | **yes**  | One of: `status`, `start`, `stop`, `restart`, `pause`, `unpause`, `logs`. |
| `path`   | string | no       | Path to the GFS repository root (instance resolved from config). |
| `id`     | string | no       | Override container name or id. |
| `logs_tail`   | number | no  | For `logs`: max lines (default: all). |
| `logs_since`  | string | no  | For `logs`: RFC3339 timestamp. |
| `logs_no_stdout` | bool | no | For `logs`: exclude stdout. |
| `logs_no_stderr` | bool | no | For `logs`: exclude stderr. |

**Returns**: Action-specific result (e.g. status object, log lines). JSON. Errors (e.g. container not found) as structured error.

---

### 8. `config` (optional)

Read or set repo config (e.g. user name/email, mount point). Equivalent to `gfs config` (see [006-cli-config](006-cli/006-cli-config.md)). May be deferred to a later iteration.

---

## Data flow and architecture

- **Context**: The MCP server is started with an optional **repository path** (e.g. env var or config file). Tools that accept an optional `path` argument use it when provided; otherwise they use this default. If no path is configured and no `path` is passed, the server may use the process cwd at startup (with clear documentation).
- **Use cases**: Each tool maps to one or more domain use cases (e.g. `status` → status use case; `commit` → commit use case). The MCP adapter must not contain business logic; it only parses tool arguments, calls the appropriate use case with the resolved repo path and options, and serializes the result (or error) for the client.
- **Ports**: Use cases depend only on domain ports (Repository, Compute, Storage, etc.). The MCP server runs in the same process as the adapters (e.g. GFS repo on disk, Docker compute); it does not introduce new ports.
- **Errors**: Domain and adapter errors are translated into MCP tool errors with a clear message and, when useful, a machine-readable code or details field.

---

## Transport and deployment

- **Transports**: The server supports:
  - **stdio (default)**: When the binary is run with no arguments (e.g. by Cursor), it uses stdin/stdout for JSON-RPC. The client spawns the server and communicates over stdio.
  - **HTTP**: When run with `--http [PORT]`, the server listens on `127.0.0.1:PORT` (default 3000) and exposes the MCP endpoint at `/mcp` (streamable HTTP/SSE). Used for daemon mode (`gfs mcp start` spawns the server with `--http`).
- **Implementation**: The workspace depends on `rmcp` with `server`, `transport-io`, and `transport-streamable-http-server`. The MCP server is a binary under `crates/data-plane/mcp`; the CLI can start it as a daemon with HTTP transport.
- **Configuration**: Repo path and other defaults may be read from environment variables (e.g. `GFS_REPO_PATH`) or a small config file so that AI assistants can point at a specific repo without passing `path` on every call.

---

## Security and safety

- **Secrets**: Status and other responses may include a **connection string** (with credentials). The MCP server MUST NOT log or expose these beyond the tool response; only the caller (e.g. the AI client) receives the result. Consider omitting or redacting the connection string in tool descriptions or logs.
- **Scope**: Tools affect only the repository and compute instance at the configured or provided path. The server does not discover or mutate arbitrary paths unless explicitly given.
- **Idempotency**: Where applicable (e.g. `status`, `log`, `list_providers`), tools are read-only and idempotent. Mutating tools (`commit`, `checkout`, `init`, `compute start/stop/...`) must be clearly named and documented so that clients can prompt or confirm before use.

---

## Out of scope

- **Control plane**: Enrollment, heartbeat, or reporting to a control plane are not part of this RFC; they remain in their respective RFCs (e.g. 003, 004).
- **Multi-repo**: A single MCP server instance is assumed to target one default repo (or one per `path` argument). Multi-tenant or multi-repo discovery is not in scope for this RFC.
- **Resources**: MCP resources (e.g. “read file at path”) are not defined here; only tools are. Resources may be added in a future RFC if needed.
- **Prompts**: MCP prompts are not defined in this RFC; the focus is on tools for GFS operations.

---

## Summary

| Item | Decision |
|------|----------|
| Purpose | Expose GFS/data-plane operations as MCP tools for AI assistants and automation. |
| Tools | `list_providers`, `status`, `commit`, `log`, `checkout`, `init`, `compute`; optional `config`. |
| Semantics | Each tool mirrors the corresponding CLI command; see RFC 006 sub-documents. |
| Repo path | Optional per-tool `path`; default from server config or cwd. |
| Architecture | MCP server as adapter; domain use cases and ports unchanged. |
| Transport | stdio (default when run by client) and HTTP (e.g. `--http 3000`) for daemon mode. |
| Secrets | Connection strings and credentials not logged; only returned in tool response when applicable. |
